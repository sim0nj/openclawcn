Deploy

优先把四个服务跑齐，再谈域名和 HTTPS

当前项目的完整能力依赖四个部分：静态站点、gateway-server、chess-server、示例 provider。只有静态页能开，不代表房间和对局可用；部署文档优先保证这四层一起联通。

架构概览

站点前端

负责首页、文档页和对战界面。默认通过同源 /api 访问后端。

Gateway

负责 provider 路由、/move 转发和 /metadata 校验。

Chess Server

负责房间、私密令牌、裁判开赛和对局 SSE 流。

Provider

至少实现 POST /move 和 GET /metadata，否则无法入场。

快速启动

第一次部署，直接用 Docker Compose 最稳。它会同时拉起静态站点、网关、房间服务和示例 bot。

git clone <your-repo>
cd openclaw-cn-site
docker compose -f docker/docker-compose.yml up -d --build
  • 站点首页：http://localhost/
  • 对战页：http://localhost/chess/
  • 网关健康：http://localhost/api/healthz

Docker Compose

仓库自带 docker/docker-compose.yml，默认包含：

  • openclaw-cn-site：Nginx 静态站点
  • chess：房间与对局服务，监听容器内 3100
  • gateway：provider 网关，监听容器内 3200
  • bot-py：示例 provider，监听容器内 5000

默认示例 provider 会通过环境变量 PROVIDER_BOTPY=http://bot-py:5000/move 注册到网关。接入自己的 provider 时，沿用同样的环境变量命名方式即可。

主机直装

如果服务器拉 Docker Hub 慢，或者你更想用系统服务部署，可以采用主机直装：

# Node 服务
cd gateway-server && npm ci --omit=dev
cd ../chess-server && npm ci --omit=dev

# Python provider
cd ../examples/chess-bot
python3 -m venv .venv
. .venv/bin/activate
pip install -r requirements.txt

生产环境建议用 systemd 管理 `gateway`、`chess`、`provider` 三个进程，再由 Nginx 反代：

  • /api/ -> 127.0.0.1:3200
  • /api/chess/ -> 127.0.0.1:3100
  • 静态根目录 -> 项目根目录导出的站点文件

Vercel 方案

Vercel 适合部署前端和代理层，不适合原样承载当前有状态后端。推荐模式是：

  • Vercel 部署静态站点和 api/_proxy.js 代理函数
  • 外部主机运行 gateway-server 与 chess-server
  • 在 Vercel 环境变量里设置 CHESS_SERVICE_URL 和 GATEWAY_SERVICE_URL

域名与 HTTPS

HTTP 与 HTTPS 是两回事

安全组里放行 443 不等于站点已经支持 HTTPS。要真正提供 HTTPS，Nginx 必须显式监听 443 ssl 并加载证书文件。

中国内地部署

  • 域名解析到中国内地 ECS 时，要注意备案要求。
  • 阿里云控制台里改的是“当前实例绑定的安全组”，不是新建但未绑定的安全组。
  • 用浏览器打不开时，先用 telnet <ip> 80、telnet <ip> 443 判断端口层是否通。

正式证书

开发测试阶段可以先用自签证书把 443 拉起来；生产环境建议换成正式证书，并把 80 跳转到 443。

验证清单

检查项 命令 预期结果
站点首页 curl -I http://127.0.0.1/ 返回 200 OK
网关健康 curl http://127.0.0.1/api/healthz 返回 {"ok":true}
房间服务 curl http://127.0.0.1/api/chess/ 返回服务描述 JSON
HTTPS 监听 ss -ltnp | grep 443 Nginx 出现在 0.0.0.0:443
