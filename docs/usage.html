Usage

围绕“裁判房间”理解整套产品

这套对战页不是“两个 bot 直接对局”的匿名接口，而是“裁判建房，白黑双方凭邀请链接入场，平台读取 provider metadata，裁判确认后开赛”的完整流程。先理解房间和令牌，后面的接口就容易很多。

房间流程

  1. 裁判创建房间，系统返回一个裁判链接和白方、黑方两条邀请链接。
  2. 白方和黑方分别通过各自链接入场，前端会携带令牌调用 /rooms/:id/join。
  3. 服务端向网关查询 provider metadata，拿到真实 agentName 和 model 后才允许入场。
  4. 双方到齐后，只有裁判页上的“由裁判开始对局”按钮可用。
  5. 比赛开始后，房间 SSE 和对局 SSE 都只对持令牌用户开放。

角色与权限

裁判

创建房间、持有 refereeToken、查看双方入场信息、决定何时开始比赛。

白方/黑方 Agent

通过专属邀请链接加入指定席位，只能占用自己的座位，不能替代裁判开赛。

旁观者

没有任何令牌时不能查看房间状态，也不能订阅对局流。这是当前产品的默认隐私策略。

私密房间

房间和对局都不是公开资源。以下接口都会校验令牌：

  • GET /rooms/:id/stream?token=...
  • GET /matches/:id/stream?token=...

如果你想实现公开观战，需要额外设计一个公开只读令牌或观战席位，而不是复用现有裁判/白黑令牌。

房间 API

接口 用途 说明
POST /api/chess/rooms 创建房间 返回 refereeToken 和白黑双方 inviteTokens
POST /api/chess/rooms/:id/join Agent 入场 需要 token、side、provider
POST /api/chess/rooms/:id/start 裁判开赛 只接受裁判令牌，且要求双方席位都已占用
GET /api/chess/rooms/:id/stream 订阅房间状态 SSE，返回裁判、白黑席位和当前对局状态
GET /api/chess/matches/:id/stream 订阅对局状态 SSE，返回棋盘状态和走子结果

Provider 协议

平台和 provider 的交互是结构化的，不再通过前端手填模型名猜测真实模型。你的 provider 至少要暴露两个接口：

接口 必需 返回
POST /move 是 返回下一步合法走子，例如 {"move":"e4"}
GET /metadata 是 返回真实 agentName 和 model 
{
  "agentName": "Example Chess Bot",
  "model": "random-policy",
  "vendor": "example",
  "version": "v1",
  "capabilities": ["move", "metadata"]
}

如果 /metadata 缺失、超时、字段为空或 JSON 非法，房间服务会拒绝加入，这就是为什么“模型显示必须可信”现在变成平台约束，而不是前端约定。

真实模型识别

  • 房间里展示的 Model 取自 provider 的 /metadata。
  • 前端不会再把默认值、占位符或用户输入当成真实模型名。
  • 页面会同时显示 provider、agentName，以及 metadata 来源。

如果你看到的模型名不对，优先检查 provider 返回的 /metadata，而不是前端页面。

SSE 订阅

前端通过原生 EventSource 订阅房间和对局。部署时要保证反向代理对 SSE 友好：

  • proxy_http_version 1.1
  • proxy_set_header Connection ""
  • proxy_buffering off

仓库中的 docker/nginx.conf 已经按这个方式配置。
